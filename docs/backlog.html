PhotoBrain — Audit & Backlog

Produced 11 Jul 2026 · full code audit (all ~4,900 lines) + live run of the app through scan → review → apply → undo + market research on competitors and distribution. Updated as implementation lands; green SHIPPED badges are verified completed work.

The one-paragraph verdict: PhotoBrain's engine is genuinely good — clean architecture, a working end-to-end pipeline, and a feature (face/expression-aware "best shot" picking) that no free Windows tool offers today. The gap between here and a tool strangers love is not missing features; it's (1) a small set of data-safety bugs that could lose a user's decisions or files, (2) a scoring formula whose face/expression intelligence is mathematically drowned out — so the headline differentiator doesn't actually decide picks yet (§F) — and a curation flow that auto-keeps all standalone junk (§G), (3) a distribution story that currently ends at a SmartScreen warning and a "Private — all rights reserved" license on a public repo, and (4) first-ten-minutes polish. Fix safety first, then make the core promise true, then launch-readiness, then delight, then scale.

What's already strong (keep and protect)

  • Architecture discipline is real. app/core/ has zero Qt imports; every pipeline stage is a pure function with progress/cancel callbacks — this is exactly the shape that makes testing and multiprocessing easy later.
  • The safety design is right: DELETE goes to Recycle Bin via send2trash (with the Windows path gotcha handled), KEEP/ARCHIVE are reversible moves, and the apply confirmation dialog changes tone when deletes are present.
  • Smart dedup optimization: pHash, scoring and face detection run once per unique SHA256 and copy to duplicates.
  • Deterministic everywhere — sorted inputs, filepath tiebreakers — so results are reproducible.
  • Live scan stats (13 counters + elapsed + current file) are far better feedback than a bare progress bar.
  • Verified live: scan → cluster → suggest → apply → undo all worked on a 15-photo synthetic set: exact dupes caught, the blurry frame archived, the sharpest kept, all 15 files restored to their original paths on undo, and a corrupt JPEG didn't crash the scan.

C. Scalability findings

The README honestly caps the app at ~5,000 photos. Real libraries people will bring are 20k–300k. digiKam's duplicate search taking "literally days" on 286k images is the cautionary tale — these four items decide whether PhotoBrain can credibly say "large batches" in its marketing. All fixes are local-only (no cloud), per the project's rules.

D. Adoption & distribution (the "make it popular" work)

Market research summary: every free Windows tool (Czkawka, dupeGuru, AntiDupl, VisiPics) picks "keepers" by file size/resolution/date — none looks at faces, open eyes, or smiles. Everything that does is $249 (Excire), subscription (Aftershoot/Narrative), or Mac-only (PhotoSweeper). "Free, local, open-source best-shot picking for Windows" is an empty niche. Czkawka's rise (~20k+ GitHub stars) came from exactly the communities PhotoBrain suits: r/DataHoarder, r/opensource, Show HN — and its stated appeal is the pitch PhotoBrain already embodies: fast, private, no telemetry. Don't compete with Czkawka on raw scan speed; position as the review/culling layer with judgment.

F. The quality score — does the "best shot" logic actually work?

The most important product finding, and it's a math problem, not a taste call. The quality score is what makes PhotoBrain different from every free competitor — it's the reason the mediapipe face/expression engine exists. Two structural defects mean the face/expression signals rarely decide anything: (1) a scale mismatch — sharpness is unbounded while every human-facing term is capped at ≤0.13; and (2) exposure leaks into "sharpness" — Laplacian variance scales with contrast², so lighting differences masquerade as focus differences and swamp the face terms. In practice the tool is a "sharpest-frame picker" wearing a "best-shot picker" label. (Revised after a second-pass review: verdicts are per-cluster, so what matters is score deltas within a group, not share-of-total — the corrected delta analysis below still reaches the same conclusion, with better evidence.)

The delta math: what would it take for a smile to win?

Suggestions are made within a cluster (top-N by score), so the question is: how big a score delta can each signal produce between two frames of the same scene?

  • Eyes fully closed → fully open is worth 0.12. Because sharpness enters as 0.45·ln(sharp), that equals a sharpness ratio of only e^(0.12/0.45) ≈ 1.31. So a photo where someone is blinking beats the open-eyed frame whenever it is ≳31% "sharper" — and raw Laplacian variance routinely swings far more than 31% between burst frames (hand shake, micro-focus, subject motion).
  • Exposure contaminates the comparison — measured: the identical image at 55% brightness has its Laplacian variance drop from 241 → 73 (−70%; it scales as brightness², verified empirically at 100/70/55/40% → 241/118/73/39). In the live run this cost the darker frame 0.54 via the "sharpness" term but only 0.04 via the actual brightness term — the formula punished the same lighting difference 14× harder through the wrong channel. A modest exposure variation between two frames (auto-exposure flicker, cloud passing) therefore outguns every face signal combined (~0.27 realistic swing).
  • Share-of-total, for context: in the live run the sharpness term was 85–97% of every photo's score. That doesn't decide verdicts by itself (deltas do), but it does make the displayed score and cross-group comparisons meaningless — a flat gray PNG posted the highest score of all 15 photos.

Net effect: the expensive face analysis is computed, stored, shown in tooltips — and can only tiebreak when two frames happen to land within a ~±30% sharpness band and identical exposure. That's rare, so the differentiator is mostly decorative today.

G. Clustering & the end-to-end test: "distill the good photos from a vacation folder"

Verdict on the core mission: PhotoBrain currently curates redundancy, not quality. It reliably finds exact duplicates and derived copies and trims within those groups — but every photo that has no near-duplicate is auto-marked KEEP, no matter how bad it is. In the live run, the second-worst photo in the whole session (a heavily blurred frame, score 0.32) and even a corrupt, half-unreadable file both received KEEP, because each sat in its own single-photo cluster. In a real vacation folder — where most junk is standalone (blurry walking shots, accidental pocket photos, dark interiors) rather than burst redundancy — the tool's promise silently skips the majority of the junk.

What the clustering does well, and its honest limits

  • Live results were correct: all three planted groups were found (4-photo burst, 3-photo burst, 3 exact copies), exact-dup groups keep exactly one, blurred members were archived, and distinct scenes stayed apart. Union-Find with per-SHA representatives is a sound, deterministic design.
  • But the test was structurally easy — a caveat on my own audit: measured pHash distances between the derived variants were 0–2 bits (pHash is DCT-based and deliberately robust to blur and brightness), while distinct scenes sat at ≥26. Any threshold from ~3 to ~25 would have produced identical clusters. The live run therefore validates the plumbing, not the threshold.
  • Threshold 17 of 64 bits is aggressive and was tuned by eye, not by data (commit history: 8 → 12 → 17). Since pHash barely moves under blur/exposure, the variance that drove the threshold up must come from angle/motion differences between real burst frames — exactly the regime where a loose threshold plus transitive Union-Find chaining (A~B, B~C ⇒ A,B,C together even if A≁C) starts merging genuinely different photos. Over-merging is the dangerous direction: "keep top 2" then archives distinct memories. Under-merging is safe but weakens distillation (both frames auto-KEEP).
  • Time is computed but never used: EXIF event grouping exists, yet clustering compares every photo against every photo across the whole folder — day-1 beach horizons can merge with day-3 beach horizons. Real burst groups are near-identical and seconds apart; ignoring time throws away the strongest available prior (and is also why the scan is O(n²)).

CLUST-01High Use EXIF time as a clustering prior (burst detection). Only union photos whose timestamps are within a window (e.g. minutes) unless they're exact SHA duplicates; the event data is already computed. This simultaneously (a) prevents cross-day over-merges, (b) matches how real bursts occur, and (c) shrinks the comparison space from O(n²) toward O(n·k) — the same change is the first step of PERF-01's scale fix. Photos without EXIF fall back to global comparison. core/clustering.py:97-102 · core/events.pyM
CLUST-02Med Validate the threshold on real photos before launch. 17/64 bits was tuned by eyeballing synthetic-ish cases; there is no labeled validation set. Fix: label ~50 real burst groups + ~50 distinct-scene pairs from a personal library, measure merge/split rates across thresholds, pick with data; add a cluster-diameter cap so chained merges can't exceed a max distance from the representative (same mechanism as PERF-04). The end-to-end scoreboard for the mission "distill quality pictures from a big folder": redundant photos (bursts, copies) — works today, subject to the threshold caveats; picking the human-best frame within a group — broken until SCORE-01/02; catching standalone junk — missing until DISTILL-01/02; not merging distinct memories at scale — unproven until CLUST-01/02. All four are contained, local-only changes in core/, and together they make the app's actual pitch true.

The plan — four phases, in order

Sequencing logic: never invite the public to a tool that can strand their files (P0 before promotion); the launch checklist is mostly non-code and can overlap (P1); delight features drive word-of-mouth after launch (P2); scale work matters once real users bring real libraries (P3).

P0 — Make it safe ≈ 1–2 weeks

Goal: no way to lose a user's files or decisions. Exit test: kill the process mid-apply → relaunch → app reconciles and undo works; undo → re-apply works; close mid-review → verdicts survive.

  1. SAFE-01 write-ahead undo journal (the one Critical)
  2. SAFE-02 + SAFE-04 fix the applied-flag lifecycle (undo resets it; skipped clusters aren't marked)
  3. SAFE-03 persist verdicts on change/close
  4. SAFE-05 unicode-safe image reads (silently-wrong verdicts are worse than crashes)
  5. SAFE-06, SAFE-07, SAFE-08 journal-on-failure, SQLite-per-thread, new-scan confirmation
  6. TEST-01 (targets 1–2) + TEST-02 file-ops & clustering tests + CI, so P0 can't regress
  7. SAFE-09..12 hardening batch as time allows

P1 — Make it launchable ≈ 1–2 weeks, overlaps P0

Goal: a stranger can find, trust, download, and successfully run it. Exit test: fresh Windows VM, download from GitHub Release, first scan of a HEIC-free folder works offline end-to-end.

  1. ADOPT-01 choose & add the license (decision needed from Nitin: MIT vs Apache-2.0)
  2. ADOPT-03 bundle mediapipe models; atomic download fallback
  3. ADOPT-02 upx off, SignPath application, winget manifest
  4. ADOPT-06 pinned requirements, opencv-headless, clean-VM smoke test
  5. SCORE-01 + SCORE-02 normalize the score terms & fix exposure handling — the differentiator ("best-shot picking") must actually work before you market it. Small, contained change; the highest-value non-safety fix in the whole backlog. DISTILL-02 (don't KEEP corrupt files) rides along for pennies.
  6. CLUST-02 validate the clustering threshold on ~100 labeled real-photo pairs — an afternoon of work that de-risks the scariest failure mode (merging distinct memories) and yields README-worthy accuracy numbers.
  7. ADOPT-04 + ADOPT-05 + ADOPT-08 README rewrite with GIF/screenshots, repo metadata, v1.0 Release, privacy-led pitch
  8. UX-14 + UX-15 settings presets & first-run explainer (the two cheapest adoptability wins in the app itself)
  9. UX-09, UX-10 (empty state), UX-11 small polish that first-time users hit in minute one

P2 — Make it lovable ≈ 3–4 weeks

Goal: the review experience itself becomes the reason people recommend it.

  1. UX-01 + UX-02 + UX-03 + UX-04 all image work off the UI thread; debounced zoom (the app suddenly feels native)
  2. FEAT-01 side-by-side compare view (flagship feature of this phase)
  3. UX-06 verdict-level undo; UX-05 shortcut discoverability; UX-07 shortcut scoping
  4. FEAT-02 HEIC/WEBP support; FEAT-03 multi-select; FEAT-04 "you'll free X GB" framing
  5. UX-08 humanize the cluster list; UX-12 colorblind badges + grid keyboard nav
  6. UX-13 QSettings persistence, one QSS theme, dark mode, DPI-aware pixmaps
  7. DISTILL-01 "low quality, no duplicates" review lane (unlocked by SCORE-01/02) — turns the tool from a duplicate-trimmer into an actual folder distiller, and no free competitor has it; DISTILL-03 quality-gap-aware keep counts
  8. CLUST-01 time-gated clustering / burst detection (also the first step of P3's scale work)
  9. SCORE-03 + SCORE-04 user-facing rating (drop the raw float) + worst-face expression penalty; FEAT-05 sort/search/progress/export; QUAL-01 face-confidence setting + boxes; ADOPT-07 launch posts once the compare view ships
  10. SCORE-05 (stretch) tunable/learned scoring; CLUST-03 "possible series" hints — the long-term moat

P3 — Make it scale ≈ 2–3 weeks, demand-driven

Goal: honest support for 50k+ photo libraries.

  1. PERF-01 numpy-popcount / BK-tree clustering (drop the "~5,000 photos" caveat)
  2. PERF-02 multiprocess the face phase; PERF-03 decode once per photo
  3. PERF-04 cluster-diameter cap (over-merge protection that also improves quality at any size)
  4. PERF-05 thumbnail draft-mode + sha-keyed cache; TEST-03 the enabling refactors along the way
  5. Then, by user demand: FEAT-06 items (multi-folder, portable mode, RAW, video)

Open decisions for Nitin

  • License (blocks P1): MIT (simplest, most contributor-friendly) vs Apache-2.0 (patent grant). Recommendation: MIT.
  • Name check before launch: "PhotoBrain" — worth a quick trademark/app-store search before investing in branding.
  • Python version target: mediapipe wheels lag CPython releases (no 3.13 at research time; this audit's venv used 3.13 successfully with the current pin — verify before pinning). Recommendation: declare 3.10–3.12 officially, test 3.13.
  • Scale ambition: if 50k+ libraries are a launch goal, pull PERF-01 into P2; if "family folder cleanup" (≤10k) is the wedge, P3 can wait for user demand.
